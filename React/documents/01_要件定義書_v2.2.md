# 施設予約システム React版 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名
汎用施設管理システム（React リライト版）

### 1.2 目的
専門学校卒業制作で作成したPHP版「汎用的施設管理システム」を、モダンな技術スタックでフルリライトする。

**システム開発の目的:**
- ユーザビリティの改善
- モジュール設計の改善
- 実務レベルのポートフォリオ作成

**技術学習の目的:**
- Next.js（App Router）の習得
- TypeScriptの習得
- Tailwind CSSの習得
- Supabaseの習得

### 1.3 システムコンセプト

```
┌─────────────────────────────────────────────────────────────┐
│            汎用施設管理システム（コア）                      │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐          │
│  │ユーザー │ │ 施設   │ │ 予約   │ │ 表示   │          │
│  │ 管理   │ │ 管理   │ │ 機能   │ │ 機能   │  ...     │
│  │モジュール│ │モジュール│ │モジュール│ │モジュール│          │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘          │
│       ON        ON        ON/OFF      ON/OFF              │
└─────────────────────────────────────────────────────────────┘
        ↓
    顧客Aには予約+表示を提供
    顧客Bには予約+料金+備品を提供
    顧客Cにはフル機能を提供
```

**ビジネスモデル:**
- 企業・顧客ごとに必要な機能モジュールをON/OFFして提供
- 必要な機能だけのパッケージとして販売可能
- 後から機能追加も容易

### 1.4 背景
- **PHP版の課題**
  - ファイル数が多く管理が困難
  - モジュール設計の問題でリファクタリングが困難
  - モバイル非対応
  - 予約フローが複雑

- **リライトで達成したいこと**
  - **モジュール設計の改善**（機能のON/OFF切り替え可能）
  - モダンなUI/UXの実現
  - コンポーネント設計による保守性向上
  - レスポンシブ対応
  - 直感的な予約フロー

## 3. モジュール構成

### 3.1 モジュール一覧

| モジュールID | モジュール名 | 説明 | デフォルト |
|-------------|-------------|------|-----------|
| M-CORE | コアシステム | 認証・基本設定（必須） | ON（固定） |
| M-USER | ユーザー管理 | ユーザー登録・認証 | ON（固定） |
| M-FACILITY | 施設管理 | 施設・カテゴリのCRUD | ON（固定） |
| M-RESERVE | 予約機能 | 予約のCRUD・重複チェック | ON/OFF |
| M-EXTEND | 時間延長 | 利用中の予約時間を延長 | ON/OFF |
| M-DISPLAY | 表示機能 | デジタルサイネージ出力 | ON/OFF |
| M-PRICE | 料金計算 | 利用料金の計算・表示 | ON/OFF |
| M-ITEM | 備品管理 | 備品の貸出管理 | ON/OFF |
| M-INQUIRY | 問い合わせ | チャット形式の問い合わせ | ON/OFF |
| M-NOTIFY | 通知機能 | メール通知 | ON/OFF |
| M-TENANT | マルチテナント | 複数組織対応 | ON/OFF |

### 3.2 モジュール依存関係

```
M-CORE（必須）
  └── M-USER（必須）
        └── M-FACILITY（必須）
              ├── M-RESERVE（任意）
              │     ├── M-EXTEND（任意）※予約がONの時のみ
              │     ├── M-PRICE（任意）※予約がONの時のみ
              │     └── M-ITEM（任意）※予約がONの時のみ
              ├── M-DISPLAY（任意）
              └── M-INQUIRY（任意）
  └── M-NOTIFY（任意）※どのモジュールとも連携可能
  └── M-TENANT（任意）※全体に影響
```

### 3.3 パッケージ例

| パッケージ名 | 対象 | 有効モジュール |
|-------------|------|---------------|
| ベーシック | 小規模オフィス | CORE, USER, FACILITY, RESERVE |
| スタンダード | 中規模企業 | + DISPLAY, INQUIRY, EXTEND |
| プレミアム | 大規模施設 | + PRICE, ITEM, NOTIFY |
| エンタープライズ | 複数拠点 | + TENANT |

## 5. ユーザビリティ改善ポイント

### 5.1 PHP版の課題と改善

| 課題 | PHP版 | React版での改善 |
|------|-------|-----------------|
| 予約状況の把握 | 一覧表形式 | **カレンダーUI**（週/月表示） |
| 予約フロー | 多ステップ | **3ステップ**（カレンダー→クリック→確認） |
| モバイル対応 | なし | **レスポンシブデザイン** |
| 空き状況確認 | ページ遷移が必要 | **リアルタイム表示** |
| モジュール管理 | 拡張機能テーブル | **設定画面でON/OFF** |

### 5.2 UI/UX設計方針

```
【予約フローの簡略化】

PHP版:
  トップ → 施設選択 → 日付選択 → 時間選択 → 入力フォーム → 確認 → 完了
  （6ステップ）

React版:
  カレンダー表示 → 空き枠をクリック → 確認モーダル → 完了
  （3ステップ）
```

### 5.3 画面設計方針

1. **カレンダー中心のUI**
   - 週表示をデフォルト
   - 縦軸：時間、横軸：施設（部屋）
   - 空き枠は色で識別（緑=空き、赤=予約済み）

2. **ワンクリック予約**
   - カレンダーの空き枠をクリック → モーダルで即予約

3. **レスポンシブ対応**
   - PC: カレンダー表示
   - モバイル: リスト表示 + 日付選択

4. **モジュール対応UI**
   - 無効なモジュールの機能は非表示
   - 管理者設定画面でモジュールON/OFF

### 7.2 PHP版からの変更点

| 項目 | PHP版 | React版 | 理由 |
|------|-------|---------|------|
| ユーザーID | varchar(30) | UUID | Supabase標準 |
| 秘密の質問 | あり | なし | メール認証に変更 |
| ログイン状態 | DBで管理 | JWTトークン | Supabase標準 |
| 拡張機能テーブル | 1テーブル | module_settings | より柔軟な設定 |
| 問い合わせ | 1テーブル | 2テーブル | チャット形式対応 |
| マルチテナント | なし | tenant_id追加 | 将来対応 |
| 予約履歴 | history_table（別テーブル） | reservations.status | シンプルな設計 |
| 時間延長設定 | extension_table | facilities.allow_extension | 施設単位で管理 |

### 7.3 予約ステータスの遷移

```
[confirmed] ──キャンセル──→ [cancelled]
     │
     │利用終了時刻経過
     ↓
[completed]
```

- **confirmed**: 予約確定状態（デフォルト）
- **cancelled**: キャンセル済み（論理削除）
- **completed**: 利用完了（end_time経過後に自動更新 or バッチ処理）

## 11. 用語定義

| 用語 | 定義 |
|------|------|
| 施設 | 予約対象となる部屋・スペース |
